Welcome to FuzzLabs

FuzzLabs in a modular fuzzing framework, written in Python. It uses a modified version of the amazing Sulley fuzzing framework (Sulley on Github) as the core fuzzing engine.

Some of the key features:

  • Modular architecture
  • Support for multiple fuzzing jobs at the same time
  • Capability of fuzzing applications over different transport mediaâ€™s (Network, File, Bluetooth)
  • Possibility to add your own transport media handlers easily (e.g. USB)
  • Web interface supporting multiple engine connections for management

Installation

Basic Setup

To start using FuzzLabs, first you have to clone the repository. Once that is done, execute the dev_setup.sh script in <FUZZLABS_HOME>. This script will install packages FuzzLabs depends on and, additional tools for development and debugging.

An additional step that might have to be done is to create the directories "queue" and "archived" under <FUZZLABS_HOME>/jobs/ if they are not there. The queue directory is where the jobs get picked up from. The archived directory contains jobs that has been finished or stopped for whatever reason.

The FuzzLabs engine can be started by issuing the following command:

/fuzzlabs.py start
The engine can be shut down with the following command:

/fuzzlabs.py stop

Web Server Setup

FuzzLabs has a web interface where you can manage your jobs and the engines as well. For this interface to be accessible you will obviously need a web server, which is basically the only thing not getting installed by the setup script. This is because you can use basically any web server you like to host the Web GUI. As the GUI is only HTML5 and JavaScript you don't need support for any dynamic languages (such as PHP). If you just do a sudo apt-get install nginx you got the web server covered.

The web server does not have to be on the same machine as the engine, but split-deployments will be discussed later. In this install guide we will be doing a deployment where all the components are on a single server.

The next step is to modify your web server's configuration to use /www as the document root. For example, in case of Nginx the following simple configuration will suffice:

server {
    listen   80;
    root /srv/FuzzLabs/www;
    index index.html;
    server_name localhost;
    location / {
        try_files $uri $uri/ /index.html;
    }
}
Once the configuration is set up, restart the web server. Then, open the GUI. With the configuration above, the URL would be simply: http://127.0.0.1/

At this point you should see the GUI displaying a form to add your first engine.

Engine Setup on the GUI

The last step of the setup process is to configure your first engine using the web interface. This you can do on the initial form which is displayed when you visit the GUI and there is no engine registered.

You have to provide the following details:

  • Engine name: This can be anything you'd like and only used to help you identify and distinguish your engines.
  • Address: The IP address or host name of the engine.
  • Port: The port number the engine is listening on. By default this is 26000.
  • Password: If the engine is protected by a password, you have to specify the password here. By default, no password is used.
There are some restrictions for some of the fields. You can read about each of these by moving your mouse over the help button next to each input field. Once you have added your first engine, you are ready to use the web interface.

Please note that all engine related data are stored in the web browser's local storage. If you clear out the local storage, all these settings are lost.
